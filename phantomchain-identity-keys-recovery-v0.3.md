## Why This Revision Exists

v0.2 derived the signing key directly from biometrics through a fuzzy extractor — "your finger *is* the key." On review, that approach is not buildable with current technology:

- A signing key requires the **exact same bits** every time, for years, across different device sensors, through aging and injury. Fingerprint/face fuzzy extractors yield only tens of bits of *stable* entropy after error correction, and only at non-trivial false-reject rates.
- A biometric-derived key has **no revocation path** — you cannot change your fingers, and re-derivation reproduces the same compromised key.
- Detecting duplicate enrollments by biometric uniqueness **contradicts** the "no biometric database" guarantee — noisy biometrics cannot be matched globally from a plain commitment hash without storing something matchable.

This revision keeps every *user-facing* promise of v0.2 — your finger authorizes everything, no seed phrase in daily use, one real human per identity, post-quantum throughout, traceable-not-surveilled — while replacing the infeasible cryptographic claim with a primitive that ships on billions of devices today.

**The reorientation:** the finger no longer *derives* the key. The finger *authorizes* a hardware-sealed key so tightly that the key is inert matter without a live biometric. Functionally, the finger and the key are one object — neither acts without the other — but the entropy comes from secure hardware, not from the biometric.

## 5. Physical Layer — Revised

### 5.1 Cryptographic Primitives

Unchanged from v0.2 except that the "Biometric KDF" row is removed — biometrics no longer produce key material.

| Primitive | Algorithm | Purpose |
|---|---|---|
| Key Encapsulation | ML-KEM (Kyber-1024) | Encrypted session establishment; recovery envelope encryption |
| Digital Signature | ML-DSA (Dilithium-3) | Transaction signing; identity-root authorization |
| Signature Fallback | SLH-DSA (SPHINCS+) | High-value signing, conservative fallback |
| Symmetric Encryption | XChaCha20-Poly1305 | TEE-sealed storage, recovery keystore |
| Hash | SHA3-256 / SHAKE-256 | Address derivation, commitment, deterministic key derivation |
| Password KDF | Argon2id | Recovery password → key material (memory-hard) |
| Threshold Signature | Dilithium-3 threshold scheme | Cluster block submission; anchor-mesh recovery share |

All primitives are NIST 2024 finalized standards plus Argon2id (RFC 9106) for the recovery path. No ECDSA. No RSA. No secp256k1.

### 5.2 Biometric-Bound Hardware Key

The day-to-day signing key is a **biometric-bound hardware key**. It is generated by the secure element's hardware RNG, sealed so it is non-exportable, and gated by a hardware-enforced policy that makes it physically unusable without a live, locally-verified biometric match.

```
Device secure element (SE-class hardware)
      │
      ▼
Hardware RNG generates Dilithium-3 keypair
  ├──► Private key:  sealed in SE, NON-EXPORTABLE, no API returns it
  └──► Public key:   exported → registered as a device credential (Section 7.3)
      │
      ▼
Usage policy enforced IN HARDWARE:
  ├──► Signing requires a fresh, live biometric match (fingerprint / face)
  ├──► Match is performed inside the SE — app never sees the template
  └──► Key is auto-destroyed if the device's biometric enrollment set changes
```

**Reference implementation surfaces (both shipping today):**

| Platform | Mechanism |
|---|---|
| Android | `KeyGenParameterSpec` with `setUserAuthenticationRequired(true)`, `setInvalidatedByBiometricEnrollment(true)`, `setIsStrongBoxBacked(true)`, `setUserAuthenticationParameters(0, AUTH_BIOMETRIC_STRONG)` |
| iOS | Secure Enclave key (`kSecAttrTokenIDSecureEnclave`) with `SecAccessControl` flags `.privateKeyUsage` + `.biometryCurrentSet` |

The finger does not derive the key. The finger unlocks the key at the hardware level so tightly that, without it, the key is inert. "Your finger is your wallet" is preserved — at the authorization layer, where it is true and enforceable.

### 5.3 Hardware Requirements

The trust root is a **dedicated secure element**, not the application TEE. Secure Enclave (Apple), StrongBox / Titan M (Android) are separate silicon from the app processor and have a materially stronger track record than general-purpose TrustZone TEEs.

- **Required:** SE-class secure element + a strong biometric sensor (fingerprint and/or face with liveness).
- **High-value tier:** signing keys for large balances are split across the user's devices via threshold/MPC signing (see Section 7.4), so no single secure element holds a complete spending key.
- Multi-vendor support is retained — no single chip manufacturer compromise affects the whole network.

### 5.4 Anti-Coercion & Duress Properties

The hardware binding yields two protections as side effects:

- **Enrollment-tamper self-destruct.** Because the key is invalidated on any change to the biometric enrollment set, an attacker who adds their own fingerprint to a seized, unlocked device **destroys** the key rather than gaining the ability to sign. Tampering bricks the key; it does not unlock it.
- **Duress finger.** A specific enrolled finger may be mapped to a duress action (silent alarm, decoy account, or local lock). This realizes the "duress finger signal" referenced in v0.2 §15.

### 5.5 Privacy Guarantees

Stronger than v0.2, because no helper data is stored:

- The biometric template never leaves the secure element and is not readable by the application.
- Matching occurs inside the secure element; no biometric data is transmitted or stored off-device.
- **No central biometric database. No biometric escrow. No stored fuzzy-extractor helper data** (the v0.2 helper data, which leaked entropy, is eliminated entirely).
- The network sees only public keys and address hashes.

### 5.6 What the Physical Layer Owns Permanently

- The secure-element-sealed, biometric-bound Dilithium-3 device keypair — non-exportable.
- The hardware attestation key — chip-embedded root.
- The attestation handshake format — fixed, backward compatible.
- Address derivation scheme — SHA3-256 of a Dilithium-3 public key.

Note what the physical layer **no longer** owns: it does not own *identity*. Identity moved up to the server layer (Section 7) so that keys can rotate without identity loss. The signing root is immutable; the binding of identity to any particular key is not.

## 7. Identity & Keys — Revised

### 7.1 Identity Is Not the Key

In v0.2, identity *was* the biometric-derived key — which is why a lost device or a compromised key meant a lost identity. v0.3 separates them:

> **Identity is a durable on-chain account. A device key is a rotatable credential the account authorizes.**

A device key can be destroyed (new phone, injury-forced re-enrollment, theft) without affecting identity or balance. This is what makes revocation possible at all.

### 7.2 The Account Record

Each identity is an on-chain account keyed by a stable **identity ID** = `SHA3-256(identity_root_public_key)`.

```
Account record (on-chain, server layer):
  {
    identity_id:            SHA3-256(identity_root_pubkey),
    identity_root_pubkey:   Dilithium-3 public key   (authority for rotation),
    authorized_device_keys: [ Dilithium-3 pubkey, ... ]  (current signers),
    guardian_set:           [ identity_id, ... ]      (NFC trusted contacts),
    recovery_policy:        { mesh_threshold: M-of-N, ... },
    balance:                attributed to identity_id, not to any device key
  }
```

Balance is attributed to `identity_id` and is invariant across device-key rotations and hardware migrations — preserving the v0.2 property that balance follows identity, not software or device.

### 7.3 Device Credentials

The public key of each biometric-bound hardware key (Section 5.2) is registered to the account as an **authorized device credential**. The network accepts a transaction only if it is signed by a device key currently listed in `authorized_device_keys`.

- One identity may authorize multiple device keys simultaneously (phone + laptop).
- Each device key is independently biometric-bound on its own secure element.
- A device key is **per-device** — the same finger on a new phone produces a *different* key (entropy is from that chip's RNG, not the finger). This is the deliberate, accepted consequence of hardware binding, and is exactly why the recovery path (Section 8) exists.

### 7.4 Key Rotation & Revocation

Revocation is a first-class operation, available through two authorities:

```
ROTATION / REVOCATION can be authorized by EITHER:

  (A) The identity-root key
      └──► reconstructed during recovery (Section 8) — used when the
           current device is lost or its key is compromised

  (B) A guardian threshold
      └──► NFC trusted contacts attest a rotation (Section 8.6)

Effect of a rotation transaction:
  ├──► add new device key to authorized_device_keys
  ├──► remove (revoke) the old/compromised device key
  └──► finalized via aBFT — immediate, irreversible
```

**High-value tier (MPC):** for large balances, the spending key is threshold-split across the user's enrolled devices. A single secure-element compromise yields one share, not a spending key. This demotes the secure element from "the trust root" to "one of N."

### 7.5 Transaction Tiers by Modality

Unchanged in spirit from v0.2 §7.6 — thresholds are server-layer parameters:

| Transaction Type | Requirement |
|---|---|
| Standard transaction | Biometric-gated device-key signature |
| High-value (above threshold) | Biometric + MPC threshold across devices |
| Key rotation / revocation | Identity-root key OR guardian threshold |
| Hardware transfer (recovery) | QR + password + anchor-mesh share (Section 8) |
| Estate beneficiary designation | Biometric + on-chain signature |

## 8. Device Recovery — Revised

### 8.1 Two Paths, Different Properties

| Path | Used For | Security Model |
|---|---|---|
| **Hot path** (Sections 5, 7) | Daily signing on a provisioned device | Biometric-bound, non-exportable secure-element key |
| **Cold path** (this section) | Migrating identity to *new* hardware after loss | QR-encrypted recovery seed, unlocked by password **+** anchor-mesh share |

The cold path is **recovery only**. It is never used for routine signing. It exists because the hot key is per-device and non-exportable, so identity must be re-bound to a fresh device key after device loss.

### 8.2 The Recovery Seed

The QR does **not** contain the hot device key (which is non-exportable and cannot be copied). It contains an encrypted **256-bit recovery seed** from which the **identity-root keypair** is deterministically derived:

```
identity_root_keypair = Dilithium3.KeyGen( SHAKE-256("phantom-root" ∥ recovery_seed) )
```

- The identity-root key's *only* power is to authorize device-key rotations (Section 7.4). It cannot itself spend in routine flow.
- It is never kept hot. It exists only as the encrypted QR payload plus the mesh share, and is reconstructed transiently inside a secure element during recovery, then discarded.
- A 256-bit seed keeps the QR payload small enough for a **single static QR frame**. (Larger payloads, if ever required, use animated multi-frame QR — proven in air-gapped hardware wallets.)

### 8.3 QR-Encrypted Keystore

```
QR payload (encrypted keystore):
  {
    version:     0x03,
    kdf:         "argon2id",
    kdf_params:  { mem, iterations, parallelism, salt },
    aead:        "xchacha20-poly1305",
    nonce:       <96/192-bit>,
    ciphertext:  Enc(K, recovery_seed),
    mesh_ref:    <anchor-mesh share locator>,
    identity_id: SHA3-256(identity_root_pubkey)
  }
```

The QR is a **copyable bearer artifact** — it can be photographed or backed up. Therefore the password alone must never be sufficient to decrypt it. That is the purpose of the split in Section 8.4.

### 8.4 Split-Key Construction

The decryption key is derived from **two factors that must both be present**: a password the user knows, and a share the anchor mesh releases only when it authorizes a transfer.

```
K       = combine( K_pw, K_mesh )
K_pw    = Argon2id(password, salt)              ← derived on-device, memory-hard
K_mesh  = threshold share released by the anchor mesh (M-of-N)
          ONLY after it authorizes the transfer
```

Consequences:

- **A captured QR + cracked password is useless offline** — `K_mesh` is missing, and the mesh rate-limits and can lock the identity. No offline brute-force is possible.
- **The mesh never sees `recovery_seed`, the password, or `K_pw`** — it holds only its share. It cannot reconstruct the wallet on its own; it can only authorize. No honeypot of recoverable wallets exists.
- **Threshold M-of-N across anchor nodes** (not a single server) means recovery is neither a single point of failure nor a single point to coerce or seize — consistent with v0.2 §12.

### 8.5 Hardware Transfer Flow

```
STEP 1 — NEW DEVICE PREP
  New hardware generates its own biometric-bound device keypair
  (Section 5.2) inside its secure element — non-exportable.
  New device displays its PUBLIC device key as a (plain) QR.

STEP 2 — USER PRESENTS RECOVERY QR
  User scans/loads their recovery QR  →  device holds ciphertext C (locked).

STEP 3 — PASSWORD PROMPT
  User enters recovery password  →  device derives K_pw.
  (Still cannot decrypt — K_mesh is absent.)

STEP 4 — SERVER / MESH AUTHORIZES TRANSFER
  Anchor mesh verifies:
    ├──► one-active-device / migration policy
    ├──► rate limit + cooldown (anti-brute-force, anti-replay nonce)
    ├──► re-prompt / confirmation
    └──► releases K_mesh (M-of-N threshold)

STEP 5 — RECONSTRUCT & RE-BIND
  Device combines K_pw + K_mesh → K → decrypts recovery_seed
  Derives identity_root_keypair INSIDE the secure element
  identity-root signs a rotation tx:
    ├──► adds the new device key to authorized_device_keys
    └──► revokes the lost device's key
  recovery_seed and identity_root_keypair are wiped from memory.

STEP 6 — RE-ISSUE
  A fresh recovery QR is generated (new salt, new mesh share split).
  The OLD recovery QR is permanently invalidated on-chain.
  Balance and identity are fully restored on the new hardware.
```

After Step 6, any previously photographed or leaked copy of the old QR is dead.

### 8.6 NFC Guardian Recovery (alternative path)

The v0.2 NFC trusted-contact flow is retained as an **alternative** rotation authority for users who lose the recovery QR or password. A guardian threshold (default 1-of-N, per v0.2 §8.3) can authorize a device-key rotation by attesting on an attested device. The two cold paths — QR keystore and guardian threshold — are independent; either can re-bind identity to new hardware.

### 8.7 Security Floor Invariant

Because the cold path is the *only* way to recover an identity, it is the **security floor of the entire system** — an attacker attacks the easiest door, and this is it. The following are therefore mandatory invariants, not optional hardening:

- A captured recovery QR **must remain useless without both** the password **and** a fresh mesh authorization.
- `K_pw` **must** use a memory-hard KDF (Argon2id) with parameters tuned so offline guessing is expensive even if the mesh share were ever to leak.
- The mesh **must** rate-limit and support identity lock on anomalous recovery attempts.
- A strong-password policy is enforced at recovery-seed creation time.

The recovery path's security must hold at the same level as the hot path. If it is weaker, it defines the real security of the system.

### 8.8 Interim Status & Deprecation Trigger

The QR recovery path is an **explicitly interim measure**. It exists because the v0.2 goal — a finger reconstructing the wallet on any device — requires biometric technology that does not yet exist. It reintroduces a memorized secret (the password), which is acceptable only because that secret is insufficient on its own (it requires the QR *and* mesh authorization).

Recovery lives entirely in the **server layer**. The signing root (Section 5) is immutable; the recovery mechanism is swappable via governance. When biometrics advance, the QR path is retired with a server-layer change — no hard fork.

> **Deprecation trigger:** The QR recovery path is retired when a biometric extractor can **reproducibly regenerate a stable ≥128-bit key from a finger across distinct devices at an acceptable false-reject rate** — i.e., when "the finger reconstructs the wallet anywhere" becomes real. Until then, QR-plus-mesh recovery is the supported migration path.

This turns the limitation into a documented, bounded tradeoff with a defined exit.

## 8A. Open Problems — Identity Layer

### Critical — Must Solve Before Mainnet

**[OPEN-ID-01] Sybil resistance / one-person-one-identity without a biometric database**
This is the one infeasible v0.2 claim this revision does **not** resolve. Biometric-bound hardware keys are per-device and never leave the device, so the network cannot tell that two devices belong to the same human. Identity-weight Sybil resistance therefore needs its own mechanism. Candidate directions:

- *ZK proof over a state identity credential* (NFC e-passport / national eID) — proves uniqueness and yields only a nullifier; PhantomChain stores no biometric. Excludes the undocumented; outsources uniqueness to states.
- *Proof-of-personhood protocols* (synchronized-ceremony or social-graph based) — inclusive but weaker than hardware-rooted uniqueness.
- *Privacy-preserving biometric dedup under MPC/HE* — honest about being an encrypted, distributed database; heavy.

A decision here is a prerequisite for the v0.2 identity-weight consensus model (§9.4) to be Sybil-resistant. Until resolved, identity weight should be treated as soft.

### Important — Before Public Launch

**[OPEN-ID-02] Anchor-mesh recovery availability vs. capture**
The M-of-N mesh share gates recovery. Parameters must balance availability (recovery still works if some anchors are offline) against capture resistance (no coercible subset can authorize a transfer). Formal threshold selection and geographic/jurisdictional diversity requirements needed.

**[OPEN-ID-03] Recovery password UX and floor strength**
The recovery password is the system's floor (Section 8.7). Real-world password strength is poor. Needs: enforced strength policy, optional passkey/second-device factor, and modeling of the offline-attack cost under a worst-case mesh-share leak.

### Research — Long-Term

**[OPEN-ID-04] Retirement path for the interim QR recovery**
When the Section 8.8 deprecation trigger is met, design the migration from QR-plus-mesh recovery to direct biometric reconstruction without disrupting existing identities.

## 8B. Changes From v0.2

| v0.2 (infeasible) | v0.3 replacement |
|---|---|
| Finger derives the signing key (fuzzy extractor) | Biometric-bound, non-exportable secure-element key — finger *authorizes*, hardware RNG supplies entropy |
| Stored fuzzy-extractor helper data | Eliminated — no helper data, no entropy leak |
| Identity = the biometric key (no revocation) | Identity = on-chain account; device keys are rotatable credentials |
| Lost device / compromised key = lost identity | Rotation/revocation via identity-root key or guardian threshold |
| One-person-one-device via biometric commitment | **Unresolved — moved to [OPEN-ID-01]** (Sybil layer) |
| TEE as sole trust root | SE-class hardware + MPC tiering; recovery via mesh threshold, not one chip |
| (No portable recovery) | Interim QR-encrypted recovery seed, password + anchor-mesh share, server-mediated, explicitly deprecatable |

All other v0.2 sections (consensus, clusters, governance, tokenomics, estate, supply chain) are unaffected. The external transaction layer and geographic coverage premium specs remain consistent — they reference identity by `identity_id`, which this revision preserves.

## 8C. Cryptographic Standards Addendum

| Standard | Algorithm | Publication | Use in this revision |
|---|---|---|---|
| ML-DSA | Dilithium-3 | FIPS 204 | Device keys, identity-root key |
| ML-KEM | Kyber-1024 | FIPS 203 | Recovery envelope / mesh share encryption |
| SHA3-256 / SHAKE-256 | — | FIPS 202 | Address + deterministic seed-to-key derivation |
| XChaCha20-Poly1305 | — | RFC 8439 (extended) | Recovery keystore AEAD |
| Argon2id | — | RFC 9106 | Recovery password KDF (memory-hard) |

No ECDSA. No RSA. No secp256k1. The recovery path adds Argon2id and reuses existing PQ primitives; it introduces no classical signing.
